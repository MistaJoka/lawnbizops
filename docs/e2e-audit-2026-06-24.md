# E2E audit — lead → done — 2026-06-24

How fast can one customer travel **lead → quoted → scheduled → worked → invoiced →
paid**, touching every feature? This logs the gaps, the wasted taps, what's
missing, and the UI issues — then a prioritized plan.

**Method.** Drove the running app in `VITE_DEMO=1` mode (in-memory "Apex Lawn &
Landscape" seed) plus a source-level read of every critical-path screen to count
exact taps and field requirements. Viewport: phone (the target Galaxy S26 Ultra).

**One-line verdict.** The data model and write path are solid; the *flow* is not.
Each screen does its one job well but **dead-ends** — after every action the user
is dumped back with no "what's next," forcing them to leave, re-navigate, and
**re-enter context the app already knows**. Worst-case lead→paid is **~30–40 taps
across 8–10 screens**. Half of those are recoverable.

---

## 1. The happy path, as tested (tap counts)

| Step | Screen | Taps now | Floor | Main waste |
|---|---|---:|---:|---|
| Create lead | `/clients/new` | 5 | 3 | no "add property" next |
| Add property | `/properties/new` | 11 | 4 | full address typed by hand; no next-step |
| Create estimate | `/money` → estimates tab → `/estimates/new` | 12–15 | 6 | **client + property re-selected**; price retyped |
| Send + accept | `/estimates/$id` | 2–3 | 2 | — |
| Schedule / job | `/schedules/new` or `/jobs/new` | 6–10 | 6 | reached only via property; estimate data not carried |
| Start → done | board / `/jobs/$id` | 1–2 | 1 | — |
| Invoice done jobs | `/invoices/new` | 6 | 4 | — (good: jobs pre-checked, `clientId` prefilled) |
| Record payment | `/invoices/$id` | 5–6 | 3 | method re-picked each time |
| **Total** | **8–10 screens** | **~30–40** | **~18** | |

The back half (job → invoice → payment) is already tight and uses deep-link
prefills well. **The front half (lead → estimate → scheduled) is where the time
goes**, and it's the part a busy operator does most.

## 2. Gaps & dead-ends (the headline theme: no forward momentum)

Severity: **P0** = blocks/forces re-navigation on the core flow · **P1** = real
friction · **P2** = polish.

- **[P0] A lead has no "Create estimate."** `/clients/$clientId` for a *lead*
  (James Okafor — "wants paver driveway quote") offers Call, Text, Edit, Log
  expense, Follow up, Log note, Add property, Archive — and **nothing to quote
  them**. The single most important action for a lead is missing. The operator
  must: back out → Money tab → switch to Estimates → +Estimate → **re-pick the
  client and property**. Confirmed live. (`clients/$clientId.index.tsx`)
- **[P0] Estimates live under "Money."** Estimating is a *sales* act done in
  front of a prospect, but the only way in is the Money tab (reads as
  accounting/AR) → estimates sub-tab → FAB. No entry from a client, a lead, the
  pipeline, or a FAB. (`money/index.tsx:113`)
- **[P0] New Estimate ignores context.** `validateSearch` is empty
  (`estimates/new.tsx:7`), so arriving from a client/property still forces a
  client **and** property re-selection. `jobs/new` and `invoices/new` already
  read search params — estimate is the outlier.
- **[P0] Accepting an estimate dead-ends.** After "Accepted" there's a
  "Create job" (disabled if no property) / "Convert to invoice" choice, but **no
  path to create a recurring schedule**, and if the estimate has no property the
  job path is simply blocked with no "add property" affordance.
  (`estimates/$estimateId.tsx:291`)
- **[P0] Recurring schedules are nearly unreachable.** `/schedules/new` is
  linked **only** from a property detail (`properties/$propertyId.index.tsx:126`).
  A lead accepting "weekly mow" can't be turned into a schedule without
  navigating client → property → add schedule, and the estimate's price/service
  don't carry over.
- **[P1] Property → nothing.** After saving a property you're returned to client
  detail with no "now quote / schedule / job this property" CTA
  (`properties/new.tsx:47`).
- **[P1] Job done → invoice is there, but payment isn't.** Job detail shows
  "Create invoice" when done (good, `jobs/$jobId.tsx:101`), but after creating the
  invoice there's no "record payment" handoff, and after recording payment no
  "next."
- **[P1] Pipeline is read-only.** `/pipeline` advances a stage (Lead→Quoted→…)
  but quoting to "Quoted" offers no "create estimate," and "Active" offers no
  "schedule work." Stage change is bookkeeping with no action attached.
- **[P1] Schedule edit silently destroys job customizations.** `resync_schedule()`
  drops untouched future `scheduled` jobs and regenerates; notes/checklist added
  to a future job are lost with no warning. (`supabase/migrations/0005…:147`)
- **[P2] Today board defaults to the empty QUOTE lane.** `LANES[0]='quote'`
  (`PipelineBoard.tsx:41`) so the board opens on an empty column while today's
  scheduled work sits off-screen right. The LaneNav jump tiles soften this, but
  the default landing should be the first non-empty / today's-work lane.
- **[P2] Declined/expired estimates and dormant clients have no recovery path**
  (no "renew estimate," no "reactivate client").

## 3. Click & time reduction (recover ~12–15 taps)

These are the cheapest wins — mostly prefill and defaults, no new screens.

1. **Deep-link New Estimate** like jobs/invoices do: accept `?clientId` &
   `?propertyId`, prefill and skip both selectors. (−2 taps, removes the worst
   re-entry) `estimates/new.tsx`
2. **"Create estimate" / "Schedule work" CTAs on client & property detail** and
   on the pipeline card — carrying `clientId`/`propertyId` so the target form is
   pre-scoped. (−3–4 taps + removes the "leave and rebuild" loop)
3. **Carry estimate → job/schedule.** From an accepted estimate, "Create
   schedule" / "Create job" should pass service, price, title, property as search
   params and prefill. (−3 taps, removes manual re-typing of the quoted price)
4. **Remember last payment method** (and last service) per org; default the
   payment form to it instead of always `cash`. (−1 tap every payment)
5. **Address autofill on New Property.** Even a single "address" field that
   geocodes (free Nominatim/OSM, matching the existing OSRM/Leaflet stack) would
   collapse 5 typed fields (line1/city/state/zip/lat-lng) into one and fix the
   "no pin" properties that break routing. (−4–6 taps + unlocks dispatch map)
6. **Default New Client to "lead"** when reached from the pipeline/lead context;
   it's the common case. (−1 tap)
7. **Success → next-step sheet.** After property/estimate/schedule save, a small
   "Next: Quote · Schedule · Job" action row instead of a bare back-nav. Keeps
   the operator in one continuous flow.

---

## 4. What to add

- **A global "+" / quick-create FAB** (or a 6th "More" nav target) that reaches
  New client, Estimate, Job, Invoice, Expense from anywhere. Today the only
  global create is "+ Job" on the Today screen; everything else is buried.
- **Make Estimates a first-class destination** — surfaced from the lead/client,
  the pipeline, and quick-create — not a Money sub-tab.
- **Lead → quote → win, as one guided flow.** Lead detail → "Quote" → estimate
  prefilled → on Accept, one tap to "Schedule recurring" or "Create job," which
  also auto-advances the client stage. Stage changes should be a *side effect* of
  doing the work, not a separate Pipeline chore.
- **Email/SMS send for estimates & invoices.** "Sent" is a manual status flip
  today; the actual send is assumed to happen outside the app. A "Send via
  email/SMS" action (even a `mailto:`/`sms:` deep link with the PDF link as a
  free first step) closes the loop and timestamps it.
- **Invoice aging nudges with action.** Money shows aging buckets; add a one-tap
  "send reminder" on overdue invoices (the `auto_overdue_reminder` setting exists
  but is invisible/uncontrolled in the UI).
- **Deposit / partial-bill on accepted estimates** (the seed already models a 50%
  paver deposit — INV-1041 — but there's no UI path to *create* a deposit invoice
  from an estimate).
- **"Renew estimate"** (clone to a new draft with a fresh valid-until) and
  **"Reactivate client"** for dormant.

## 5. UI / polish issues (against the project rubric)

Most screens pass the rubric — these are the gaps found:

- **Pipeline empty state** uses bare text, not `EmptyState` w/ CTA
  (`pipeline.tsx:~79`).
- **tabular-nums** not consistently applied to money columns (alignment relies on
  font, drifts on the A/R and reports tables).
- **Tiny tap targets** on photo-delete (~28px) below the 44px floor
  (`jobs/$jobId.tsx` photos, `estimates/$estimateId.tsx` photos).
- **Board default scroll** lands on the empty QUOTE lane (§2 P2).
- **No client-side validation messaging** on New Client (saves with name-only,
  no inline guidance) — minor but the only validation-less core form.
- **Discoverability**: Dispatch (map), Tools, Inventory, Tax, Reports are
  reachable only via single deep links (Today alert / Settings list), so most
  users will never find them. Consider grouping under a "More" surface.

## 6. Prioritized plan

Sequenced by impact-per-effort. Each item is small enough for one `/loop`
iteration (smallest change, full gate, push) per `docs/worklists/README.md`.
Concrete items have been appended to the relevant area worklists.

### Phase 1 — Close the dead-ends (P0, ~1–2 days)
Highest impact: turns a disconnected set of screens into a flow.
1. **Deep-link New Estimate** (`?clientId`/`?propertyId` prefill).
2. **"Create estimate" CTA on client & lead detail** (→ prefilled estimate).
3. **"Quote"/"Schedule" actions on the Pipeline card**, carrying client context.
4. **Estimate-accept → "Create schedule"** path (+ "add property" affordance when
   the job path is blocked).
5. **Carry estimate service/price/property into job & schedule forms.**

### Phase 2 — Smart defaults & prefill (P1, ~1 day)
Pure tap reduction, no new screens.
6. Remember & default **last payment method** (and last service).
7. **Next-step success sheet** after property / estimate / schedule save.
8. **Default New Client to "lead"** from lead/pipeline entry.
9. **Board defaults to today's first non-empty lane.**

### Phase 3 — New surfaces (P1–P2, ~2–3 days)
10. **Global quick-create FAB** (client/estimate/job/invoice/expense).
11. **Address autofill + geocode** on New Property (free Nominatim/OSM).
12. **Send estimate/invoice** action (mailto/sms deep link → later real email).
13. **One-tap overdue reminder** on Money aging buckets.

### Phase 4 — Recovery & polish (P2)
14. Schedule-edit **confirm dialog** when regeneration would drop customized jobs.
15. **Renew estimate** / **Reactivate client**.
16. Rubric cleanups: Pipeline `EmptyState`, `tabular-nums` on money tables, 44px
    photo-delete targets, New Client validation.
17. **Deposit invoice** from an accepted estimate.

**If only one thing ships:** Phase 1 items 1–2. "Quote this lead" from the lead's
own screen, prefilled, is the difference between the app feeling like a connected
CRM and a folder of forms.

# Appendix A — Deeper pass (features off the core path + data integrity)

Second sweep: drove the features the first pass only read (dispatch, schedule,
reports, property/estimate forms — screenshots captured), plus a correctness
audit of the lifecycle. **Claims below were re-verified against source — two
"critical double-billing" findings were overstated and are corrected here.**

## A1. Data-integrity & correctness (verified)

Severity reflects *verified* reality, not the first read.

- **[MED] No DB guard against estimate→multiple invoices.** The "Convert to
  invoice" button *is* guarded client-side (disabled once `linkedInvoiceId` is
  set — `estimates/$estimateId.tsx:219`, handler `:92`), so this is **not** an
  active double-bill. But there's **no `unique(estimate_id)` constraint** on
  `invoices`, so two offline conversions before sync, or a direct write, could
  still create duplicates. Fix = add the partial unique index (defense-in-depth).
- **[MED] Voiding an invoice ignores its payments.** `voidInvoice()`
  (`invoices/hooks.ts:485`) flips status→`void`, reverts jobs→`done`, logs an
  activity — but does **not** reverse/flag recorded `payments`. Cash-basis
  collected-revenue (reports/dashboard sum payments by `paid_at`) keeps counting
  money against a voided invoice. Re-invoicing the void's jobs is *correct*
  (it's a compensating transaction, not a double-bill) — the real gap is the
  orphaned payments. Fix = on void with payments, warn + auto-reverse or exclude
  void invoices from collected sums.
- **[MED-HIGH] Schedule edit silently destroys job customizations.**
  `resync_schedule()` (`0005:147`) does
  `delete … where status='scheduled' and scheduled_date >= current_date`. Any
  future visit you edited (notes, price, time, checklist) but didn't start is
  deleted and regenerated at the schedule's defaults. No warning. Fix = a
  `customized_at` stamp that's never resynced, or a ConfirmDialog naming the N
  affected visits.
- **[MED] Materialization horizon is 56 days** (`dates.ts` `materializeHorizon`),
  run once per app open (`index.tsx:66`). A weekly client whose operator doesn't
  open the app for 2 months has empty schedule days past the horizon — silent
  operational blindness. Fix = extend to ~6 months and/or a `pg_cron` weekly
  top-up; on load, auto-extend if `last_materialized_through < today+4w`.
- **[HIGH-UX] Outbox permanent-failure is quiet.** A 4xx parks the op as
  `failed` and **halts the FIFO** behind it (`outbox.ts`); the only surface is
  the Settings → Sync screen. A user who never opens it accumulates stuck writes
  and may never know a write was lost. Fix = a real toast/banner when an op hits
  terminal failure + a one-tap retry with the error reason inline.
- **[MED] Money basis is split and unlabeled.** Job profitability = **billed**
  (`profitability/hooks.ts`, by invoice `issued_at`); client profitability &
  dashboard = **collected** (by payment `paid_at`). Both are defensible but the
  UI never says which, so "$5,000 job / $0 client revenue" looks like a bug.
  Fix = label them "Billed" vs "Collected" in the UI.
- **[LOW-MED] Archive leaves orphans.** Archiving a client doesn't check for open
  invoices; archiving a property doesn't pause its active schedule (which keeps
  materializing jobs for a hidden property). Fix = warn / auto-pause on archive.
- **[LOW] Multi-tenant code in a single-tenant prod** — known debt per CLAUDE.md
  (RLS built, prod auth-off). Not a bug; flagged for the Phase E cutover.

## A2. Feature deep-dive (off the core path)

- **Dispatch map** (good feature): stops are labeled by **property label**
  ("Common area / Home / Driveway job") not client name — ambiguous when several
  clients have "Home" (confirmed live). And **unpinned jobs** (no lat/lng) are
  listed as plain text with **no tap target** — you can see "this job has no
  location" but can't tap to fix it. Fix = show client name on each stop; make
  unpinned rows link to the job so you can add a location.
- **Reports**: solid P&L / aging / category / method, but **job-level
  profitability data exists and is never surfaced** here (only per-client) — no
  "most/least profitable jobs or services," which is the one insight that fixes
  pricing. Empty sections use plain text, not `EmptyState`; money columns lack
  `tabular-nums`. "Cash basis" P&L sits directly above an accrual-style A/R
  aging block with no explanation (see A1 money-basis).
- **Schedule weekly view** (good): day strip + status chips + `+JOB`. Gaps:
  **no recurring-vs-one-off indicator** (can't tell a standing gig from a
  one-off), and prices aren't `tabular-nums`.
- **Tools (mulch/grade)**: results are **dead-ends** — no "add to estimate" /
  "save to job notes," so a calculated 8 cu-yd mulch volume must be retyped into
  an estimate line by hand. Make results actionable into an estimate/job.
- **Inventory**: stock badges + quick-add are nice, but **no job/expense
  linkage** (can't attribute "mulch used on the Nguyen job") and **+/- adjusts
  don't log usage history**. Quick-add has no name validation.
- **Expenses**: genuinely fast (amount→category→save, camera receipt capture,
  offline-resilient, deep-links `clientId`/`jobId`). Main gap: no reciprocal
  "expenses for this job" view until the edit screen.
- **Tax/mileage**: light logging, but **no mileage list/edit/delete** (log-only),
  and the mileage-rate / quarterly-% setup is a plain-text link that's easy to
  miss with no CTA when unset.
- **Automations**: toggles for auto-followup / auto-overdue exist but give **no
  feedback that anything runs** (no "last ran / next run," no preview), delay
  inputs are below the 44px target, and enabling has no confirm. Users can't
  tell if these are wired to a live cron.
- **Onboarding**: refreshingly short — business name (required) + phone + two
  toggles (seed 8 starter services, add a sample client), seeds the catalog. Gap:
  sample data hardcodes FL; no "what next" hand-off into the empty app.

# Appendix B — What's missing vs. real lawn-care CRMs (customer-facing half)

Benchmarked the solo/small tier of Jobber, Housecall Pro, Yardbook, Service
Autopilot, Aspire. Verdict: LawnBizOps is a polished **operator-facing ops tool**
(CRM, scheduling, dispatch/routing, invoicing, tax — genuinely strong). The
**entire customer-facing half** every competitor sells is absent. The system
boundary is the find: **every "send" today is a manual PDF to the Android share
sheet, every reminder is a `tel:`/`sms:` link the operator taps by hand — the
customer never interacts with the system.**

The encouraging part: ~70% of the high-value gaps fit the free/offline-first
constraint via device-native deep links, free email tiers, and pay-per-use
Stripe (no fixed cost). Only automated *bulk SMS* inherently needs a paid
service — and competitors gate exactly these (reminders, reviews, portal) behind
$29–119/mo upsell tiers, so this is the seam to undercut at zero fixed cost.

| Capability | Competitors | LawnBizOps | Constraint |
|---|---|---|---|
| Web "request a quote" lead form | ✅ all | ❌ | free |
| Online estimate approval link | ✅ (Yardbook free, no login) | ❌ (manual status flip) | free |
| Online payment / pay link | ✅ Stripe | ❌ (manual record only) | pay-per-use |
| Card-on-file / recurring auto-billing | ✅ | ❌ | pay-per-use |
| Appointment reminders (email/SMS) | ✅ (Jobber gates at $119) | ❌ | free email / paid SMS |
| "On my way" text | ✅ | ❌ | free (native sms:) |
| Before/after photos to customer | ✅ | ❌ (photos internal-only) | free |
| Auto Google review request | add-on $39/mo | ❌ | free |
| Customer portal | ✅ | ❌ | free |
| Time-on-site / job costing | ✅ | ❌ (`start_time` is a text field) | free |

## Build order (value ÷ effort; all free unless noted)
1. **Google review request link** — store the operator's Google `placeid`, emit
   a `writereview?placeid=` link post-job. Trivial; competitors charge $39/mo. *(free)*
2. **One-tap "on my way" SMS** — reuse the existing `cardActions.ts` `sms:`
   pattern. *(free)*
3. **Time-on-site capture** — `time_entries` driven by the existing start/done
   buttons; unlocks *real* job costing (today profitability has no labor time). *(free)*
4. **Public "Request a Quote" form** — the biggest structural gap; opens the top
   of the funnel. One public route → edge function / anon-insert → `leads`. *(free)*
5. **Online estimate approval link** — token-keyed public view + anon "Accept"
   RPC that stamps the timestamp. Yardbook proves the free no-login model. *(free)*
6. **Share before/after photos with the customer** — photos already captured;
   surface them in the shared summary/link. *(free)*
7. **Automated email** reminders / confirmations / receipts — edge function +
   Resend (3k/mo free) or Brevo. The achievable automated-comms win. *(free)*
8. **Stripe Payment Link on invoices** — no fixed cost, 2.9%+30¢ only when paid;
   webhook flips invoice→paid; degrades to manual record. *(pay-per-use)*
9. **Card-on-file + recurring auto-billing** — highest-ROI billing feature for a
   recurring-mowing business. *(pay-per-use, higher effort)*

**Do NOT build:** automated bulk Twilio SMS — A2P 10DLC registration + monthly
number + per-message carrier fees, no free tier. Substitute free email + native
click-to-text. (Matches the prefer-free rule.)

**Architectural note.** Items 4–8 need something new here: **public,
unauthenticated surfaces** (lead form, approval link, pay link, review link) —
deliberately breaking the "everything behind `_authed`" model. The offline-first
outbox still governs all *operator* writes; customer-facing views are inherently
online and sit cleanly beside the existing Stripe edge-function pattern
(`supabase/functions/stripe-*`) via token-keyed anon RLS.

# Revised plan (supersedes §6 — three tracks)

The original §6 plan (close the in-app dead-ends) still stands as **Track A** and
is still the first thing to ship. The deeper pass adds two more tracks.

### Track A — Flow & friction (in-app, P0/P1) — ship first
§6 Phases 1–2: deep-link New Estimate · "Quote/Schedule" CTAs on lead/client/
pipeline · estimate→schedule path · carry estimate data into job/schedule ·
last-payment-method default · next-step success sheet · board lands on today's
work. **Biggest bang, smallest effort. Start here.** (worklist items seeded.)

### Track B — Correctness & trust (do before/with any billing changes)
From Appendix A — these protect data and money:
1. `unique(estimate_id)` partial index on invoices (defense-in-depth).
2. Void-with-payments: warn + reverse/flag so collected revenue can't count a
   voided invoice.
3. Schedule resync: `customized_at` guard or ConfirmDialog naming affected visits.
4. Extend materialization horizon to ~6mo + auto-extend on load (and/or pg_cron).
5. Outbox terminal-failure: visible toast/banner + inline retry with reason.
6. Label money "Billed" vs "Collected" across job/client/reports/dashboard.
7. Surface job/service profitability in Reports (data already exists).
8. Dispatch: client name on stops; make unpinned jobs tappable.

### Track C — Customer-facing half (the strategic gap, sequenced)
From Appendix B build order. Sequence: review link → on-my-way SMS →
time-on-site → **public quote form** → estimate approval link → share photos →
automated email → Stripe pay link → card-on-file/auto-billing. Tracks A/B are
weeks; Track C is the multi-month roadmap that turns an ops tool into a CRM that
also wins and retains customers. Slots into the SaaS roadmap in `docs/crm-roadmap.md`.

**Sequencing call:** Track A now (cheap, daily friction). Track B before any
online-payment work (don't put money flows on shaky invoicing). Track C item 1
(review link) is a free quick win that can ship anytime. Track C item 4 (public
lead form) is the highest *strategic* value — it's the only item that grows the
business rather than streamlining existing work.
